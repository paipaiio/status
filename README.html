一套镜像，两组 Deployment，数据进 PVC

这个项目现在可以按 Kubernetes 方式部署：同一个 Docker 镜像同时包含状态监测站和 QQ 群截图机器人。K8s 里拆成 `status-site` 和 `qq-status-bot` 两个 Deployment，分别暴露服务，分别挂载 PVC。

Node.js 22 Chromium 截图 PVC 持久化 NapCat 反向 WebSocket 无前端构建步骤

状态站

`node src/server.js`，监听 `3210`，提供首页、后台、状态 API、WebSocket `/socket`。

QQ 机器人

`node src/qq-status-bot.js`，监听 `3211/onebot`，NapCat 连接后群内发送触发词返回截图。

数据存储

状态历史、后台动态模型、批注、QQ 临时截图都写到 `/data` 下，由 PVC 持久化。

文件清单

这次新增的是 K8s 部署需要的最小完整集合。

路径 用途 是否要改
Dockerfile 构建统一镜像，内置 Node.js、Chromium、中文字体。 通常不用改。
.dockerignore 排除 `.env`、运行时数据、Git、旧归档等内容，避免打进镜像。 通常不用改。
k8s/00-namespace.yaml 创建命名空间 api-relay-status。 要换命名空间时改。
k8s/01-configmap.yaml 保存非敏感配置，包括端口、路径、截图参数、状态页 URL。 生产域名和截图参数建议改。
k8s/02-pvc.yaml 创建 `status-data-pvc` 和 `qq-bot-data-pvc`。 按集群 StorageClass 和容量调整。
k8s/03-status-site.yaml 状态站 Deployment 和 Service。 需要替换镜像地址。
k8s/04-qq-bot.yaml QQ 机器人 Deployment 和 Service。 需要替换镜像地址。
k8s/05-ingress.example.yaml 状态页和 OneBot 反向 WebSocket 的 Ingress 示例。 按你的域名和 IngressClass 改。
k8s/examples/secret.example.yaml Secret 模板，放后台密码、接口 Token、OneBot Token。 必须复制后替换真实值。
k8s/examples/initial-target-secret-fields.example.yaml 可选的初始检测项字段示例。后台添加模型后会写入 PVC，不强制用 env 配。 只在想用 Secret 固定初始化检测项时参考。

PVC 数据路径

K8s 里不再依赖 `.env` 文件或本机 `runtime/`。所有会变的数据都通过环境变量指向 `/data`，再由 PVC 接住。

PVC 容器挂载点 实际文件 内容 是否敏感
status-data-pvc /data /data/config/targets.json 后台动态新增的检测模型，包含 Base URL、模型、API Key。 敏感，PVC 备份要加密。
status-data-pvc /data /data/runtime/status-history.json 状态历史记录，用于 90 分钟、24 小时、7 天、30 天可用度。 一般不敏感。
status-data-pvc /data /data/runtime/annotations.json 后台和外部接口写入的批注、维护说明、告警状态。 可能敏感。
qq-bot-data-pvc /data/qq-bot /data/qq-bot/status-*.png QQ 截图临时文件，发送后会删除。 通常不需要长期保留。
关键点

`config/targets.example.json` 仍然在镜像内，真正的后台动态数据不要挂载到 `/app/config`，而是写到 `/data/config/targets.json`。这样不会把镜像里的代码和示例配置盖掉。

配置与密钥

非敏感值放 ConfigMap，账号、Token、API Key 放 Secret。生产环境不要把真实 Secret 提交到仓库。

ConfigMap 重点项

  • FRONTEND_ORIGIN：状态页公网域名。
  • STATUS_PAGE_URL：QQ 机器人截图打开的地址，默认用集群内服务 http://status-site:3210/?capture=qq。
  • MANAGED_TARGETS_FILE：固定为 /data/config/targets.json。
  • STATUS_HISTORY_FILE：固定为 /data/runtime/status-history.json。
  • ANNOTATIONS_FILE：固定为 /data/runtime/annotations.json。

Secret 重点项

  • ADMIN_USERNAME、ADMIN_PASSWORD：后台登录。
  • MANUAL_CHECK_TOKEN：手动触发检测接口。
  • ANNOTATION_API_TOKEN：外部系统写批注接口。
  • ONEBOT_SHARED_TOKEN：NapCat 反向 WebSocket Bearer Token。
  • API_1_*：可选初始中转站配置。更推荐在后台 `/admin` 添加，写入 PVC。
k8s/examples/secret.example.yaml 
apiVersion: v1
kind: Secret
metadata:
  name: api-relay-status-secret
  namespace: api-relay-status
type: Opaque
stringData:
  ADMIN_USERNAME: "root"
  ADMIN_PASSWORD: "change-me-admin-password"
  MANUAL_CHECK_TOKEN: "change-me-manual-check-token"
  ANNOTATION_API_TOKEN: "change-me-annotation-token"
  NOTIFICATION_WEBHOOK_TOKEN: ""
  ONEBOT_SHARED_TOKEN: ""
检测项不必写在 env 里

服务可以先以 0 个检测项启动，然后登录后台 `/admin` 添加模型。后台添加的数据会写入 PVC 的 /data/config/targets.json。只有你希望 K8s Secret 在每次启动时固定注入一批初始检测项，才需要参考 k8s/examples/initial-target-secret-fields.example.yaml 添加 API_1_* 字段。

部署步骤

下面命令按“构建镜像、创建 Secret、部署 K8s、检查状态”的顺序走。

快速部署命令 
IMAGE=registry.example.com/api-relay-status:0.1.0

docker build -t "$IMAGE" .
docker push "$IMAGE"

kubectl apply -f k8s/00-namespace.yaml
cp k8s/examples/secret.example.yaml /tmp/api-relay-status-secret.yaml
vim /tmp/api-relay-status-secret.yaml
kubectl apply -f /tmp/api-relay-status-secret.yaml

kubectl apply -k k8s
kubectl -n api-relay-status rollout status deploy/status-site
kubectl -n api-relay-status rollout status deploy/qq-status-bot
kubectl -n api-relay-status get pods,svc,pvc

替换镜像地址

把 k8s/kustomization.yaml、k8s/03-status-site.yaml、k8s/04-qq-bot.yaml 里的 registry.example.com/api-relay-status:0.1.0 换成你自己的镜像。

创建 Secret

复制 k8s/examples/secret.example.yaml 到临时文件，填真实后台密码和 Token 后再 apply。检测项可以部署后从后台添加，不必在 Secret 里预先写 API Key。

部署主资源

kubectl apply -k k8s 会创建 Namespace、ConfigMap、PVC、状态站、QQ 机器人。Ingress 示例默认不在 kustomization 里，确认域名后再单独 apply。

检查健康状态

状态站探针走 /api/health，QQ 机器人探针走 /health。两个 Deployment 都 ready 后，再接 Ingress 或 NapCat。

QQ 机器人接入

NapCat 使用 OneBot 反向 WebSocket 主动连接本项目。群里发送完整触发词 `状态检查` 后，机器人打开状态页截图并发送 base64 图片。

项目 值 说明
Deployment qq-status-bot 由 k8s/04-qq-bot.yaml 创建，执行 node src/qq-status-bot.js。
Service qq-status-bot:3211 NapCat 反向 WebSocket 连接这个服务。
OneBot 路径 /onebot 由 QQ_BOT_PATH 控制，默认 /onebot。
健康检查 /health K8s readiness/liveness 都用这个 HTTP 端点。
PVC 路径 /data/qq-bot 截图临时文件 status-*.png 写在这里，发送后自动删除。
截图来源 STATUS_PAGE_URL 默认 http://status-site:3210/?capture=qq，机器人在集群内访问状态站。
截图复用 STATUS_CAPTURE_CACHE_MS=10000 多个群短时间内同时触发时复用同一批截图，避免重复拉起 Chromium。

NapCat 在同一个 K8s 集群

反向 WebSocket 地址填集群内 Service。

集群内地址 
ws://qq-status-bot.api-relay-status.svc.cluster.local:3211/onebot

NapCat 在集群外

先改好 k8s/05-ingress.example.yaml，再用公网域名连接。

公网地址 
wss://onebot.example.com/onebot
Token 配置

如果 ONEBOT_SHARED_TOKEN 非空，NapCat 也要配置同一个 access token。项目会校验 Authorization: Bearer <ONEBOT_SHARED_TOKEN>。

QQ 机器人验证命令 
kubectl -n api-relay-status rollout status deploy/qq-status-bot
kubectl -n api-relay-status logs deploy/qq-status-bot -f
kubectl -n api-relay-status port-forward svc/qq-status-bot 3211:3211
curl http://127.0.0.1:3211/health

接口调用速查

对外域名走 Ingress，本地排障可以用 port-forward。

先转发状态站 
kubectl -n api-relay-status port-forward svc/status-site 3210:3210
BASE=http://127.0.0.1:3210

公开查询

公开接口 
curl "$BASE/api/health"
curl "$BASE/api/status"
curl "$BASE/api/config"

手动触发检测

需要 MANUAL_CHECK_TOKEN 
curl -X POST "$BASE/api/check" \
  -H "X-Admin-Token: $MANUAL_CHECK_TOKEN"

curl -X POST "$BASE/api/check?target=api-1" \
  -H "X-Admin-Token: $MANUAL_CHECK_TOKEN"

后台登录 Cookie

后台接口 
curl -c /tmp/status-cookie.txt -X POST "$BASE/api/admin/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"root","password":"change-me-admin-password"}'

curl -b /tmp/status-cookie.txt "$BASE/api/admin/targets"
curl -b /tmp/status-cookie.txt "$BASE/api/admin/annotations"

外部写批注

需要 ANNOTATION_API_TOKEN 
curl -X POST "$BASE/api/integrations/annotations" \
  -H "Content-Type: application/json" \
  -H "X-Annotation-Token: $ANNOTATION_API_TOKEN" \
  -d '{
    "title": "自动告警",
    "body": "外部监控发现请求失败率升高。",
    "level": "critical",
    "targetId": "api-1",
    "notify": true
  }'

运维命令

这些命令用于看日志、确认 PVC 文件、重启和排障。

查看日志

logs 
kubectl -n api-relay-status logs deploy/status-site -f
kubectl -n api-relay-status logs deploy/qq-status-bot -f

确认数据文件

pvc files 
kubectl -n api-relay-status exec deploy/status-site -- ls -lah /data/runtime /data/config
kubectl -n api-relay-status exec deploy/status-site -- cat /data/runtime/status-history.json
kubectl -n api-relay-status exec deploy/status-site -- cat /data/runtime/annotations.json

重启服务

restart 
kubectl -n api-relay-status rollout restart deploy/status-site
kubectl -n api-relay-status rollout restart deploy/qq-status-bot
kubectl -n api-relay-status rollout status deploy/status-site
kubectl -n api-relay-status rollout status deploy/qq-status-bot

检查服务

status 
kubectl -n api-relay-status get pods,svc,pvc
kubectl -n api-relay-status describe pod -l app=status-site
kubectl -n api-relay-status describe pod -l app=qq-status-bot
